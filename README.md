### Задание 1

Опишите основные преимущества использования масштабирования методами:

- активный master-сервер и пассивный репликационный slave-сервер; 
- master-сервер и несколько slave-серверов;

- Преимущества использования масштабирования методами:

-Активный master-сервер и пассивный репликационный slave-сервер:

Отказоустойчивость: Пассивный slave-сервер выступает резервной копией данных. В случае сбоя master-сервера slave может быть быстро переведен в режим master, минимизируя простой системы.
Безопасность данных: Репликация обеспечивает резервное копирование информации, защищая от потери данных при аппаратных или программных сбоях.
Простота управления: Схема проста в настройке и поддержке, так как slave-сервер не участвует в активной обработке запросов, а лишь синхронизирует данные.
Изоляция нагрузки: Все операции записи и чтения выполняются на master-сервере, что упрощает контроль согласованности данных без риска конфликтов между репликами.

-Master-сервер и несколько slave-серверов:
Масштабируемость чтения: Slave-серверы распределяют нагрузку на чтение (например, SELECT-запросы), что значительно повышает производительность систем с высоким объемом операций чтения (например, интернет-магазины, аналитические платформы).
Высокая доступность: Даже при выходе из строя одного из slave-серверов система продолжает функционировать, так как остальные реплики остаются доступными.
Гибкость в балансировке нагрузки: Запросы на чтение можно равномерно распределять между slave-серверами, оптимизируя использование ресурсов.
Горизонтальное масштабирование: Возможность добавления новых slave-серверов позволяет гибко увеличивать пропускную способность системы под рост нагрузки.
Разделение задач: Master-сервер сосредоточен на операциях записи (INSERT/UPDATE), а slave-серверы — на чтении, что улучшает общую эффективность работы.

-Итог:
Первая схема (активный master + пассивный slave) идеальна для обеспечения отказоустойчивости и резервного копирования, но не масштабирует производительность. Вторая схема (master + несколько slave) позволяет одновременно повысить отказоустойчивость, доступность и пропускную способность за счет распределения нагрузки на чтение, что особенно важно для высоконагруженных приложений. Однако она требует внимательного управления синхронизацией данных и может вносить небольшие задержки в актуальность информации на slave-серверах.

### Задание 2


Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц: 

- пользователи, 
- книги, 
- магазины (столбцы произвольно). 

Опишите принципы построения системы и их разграничение или разбивку между базами данных.

### План горизонтального и вертикального шаринга базы данных

#### **Принципы построения системы**
1. **Горизонтальный шаринг**:
   - Разделение таблиц по **строкам** на основе логических критериев (регион, идентификатор, популярность).
   - Цель: снижение нагрузки на отдельные серверы и улучшение производительности при обработке запросов.

2. **Вертикальный шаринг**:
   - Разделение таблиц по **столбцам** на основе частоты использования данных.
   - Цель: вынос редко используемых данных в отдельные базы для оптимизации доступа.

3. **Связи между таблицами**:
   - Сохранение внешних ключей внутри одного шарда или использование сервисов синхронизации для распределенных связей.

---

#### **Разграничение таблиц между базами данных**

##### **1. Таблица "Пользователи"**
- **Горизонтальный шаринг**:
  - Разделение по **регионам** (например, Европа, Азия, Северная Америка).
  - Каждый шард содержит пользователей из одного региона.
- **Вертикальный шаринг**:
  - Основная информация: `user_id`, `name`, `email`, `subscription_type`.
  - Дополнительная информация: `user_id`, `history_orders`, `preferences`.

##### **2. Таблица "Книги"**
- **Горизонтальный шаринг**:
  - Разделение по **жанрам** (например, фантастика, детективы, образовательные).
- **Вертикальный шаринг**:
  - Метаданные: `book_id`, `title`, `author`, `genre`.
  - Контент: `book_id`, `text`, `cover_image`.

##### **3. Таблица "Магазины"**
- **Горизонтальный шаринг**:
  - Разделение по **географическим зонам** (например, города, страны).
- **Вертикальный шаринг**:
  - Контактная информация: `store_id`, `address`, `phone`, `manager`.
  - Инвентарь: `store_id`, `stock_books`, `suppliers`.

---

---

#### **Режимы работы серверов**
1. **Активный-активный**:
   - Все серверы принимают запросы одновременно (для балансировки нагрузки).
   - Пример: серверы 1–3 обрабатывают параллельные запросы от клиентов.

2. **Активный-пассивный**:
   - Резервные серверы (например, сервер 4–6) используются для репликации данных и автоматического переключения при сбое основного сервера.

3. **Репликация**:
   - Горизонтальные шарды синхронизируются через асинхронную репликацию для обеспечения консистентности.
   - Вертикальные шарды используют транзакционные журналы для синхронизации связанных данных.

---

#### **Примеры связей между шардами**
- **Пользователь → Магазин**: Используется внешний ключ `user_id` в таблице заказов, который хранится в том же шарде, что и пользователь.
- **Книги → Магазины**: Связь через `book_id` и `store_id` реализуется через API-сервисы, которые собирают данные из разных шардов.

---

#### **Дополнительные механизмы**
- **Маршрутизация запросов**: Использование прокси-сервера (например, Vitess для MySQL) для направления запросов к нужным шардам.
- **Транзакции**: Для распределенных операций (например, покупка книги) применяется двухфазный коммит (2PC) или Saga-паттерн.

Этот подход обеспечивает масштабируемость, отказоустойчивость и оптимизацию нагрузки на серверы.
